# oxlint parity notes

Record of eslint rules from the previous setup that don't have a 1:1 oxlint equivalent. JSON config doesn't support comments, so the gaps live here. Revisit when oxlint adds support.

## Replaced by oxfmt (formatter)

- `prettier/prettier` — formatting concerns moved to oxfmt; no longer enforced via lint.
- `react/jsx-curly-brace-presence` (kept, but oxfmt would also flag mechanical noise here).

## Replaced by built-in oxlint rules

- `unused-imports/no-unused-imports` and `unused-imports/no-unused-vars` — covered by `typescript/no-unused-vars` (which already understands the unused-imports plugin's behaviour).
- `@typescript-eslint/consistent-type-imports` → `typescript/consistent-type-imports`.
- `@typescript-eslint/no-explicit-any` → `typescript/no-explicit-any`.
- `@typescript-eslint/explicit-module-boundary-types` → `typescript/explicit-module-boundary-types` (kept off).
- React recommended ruleset → `react` plugin under `correctness`/`suspicious` categories.
- React-hooks recommended ruleset → also covered by the `react` plugin.

## Open parity gaps (TODO oxlint-parity)

These rules were enforced by eslint and have **no current oxlint equivalent**:

- **`import/order`** — oxlint's `import` plugin lacks the full grouped/alphabetised ordering options. Mitigation: oxfmt may handle import ordering on its own; verify in commit 11.
- **`simple-import-sort/exports`** — no oxlint equivalent. Same mitigation as above.
- **`react/jsx-uses-react`** and **`import/no-unused-modules`** — not implemented in oxlint's react/import plugins. Dropped.
- **`jsx-a11y/no-noninteractive-element-to-interactive-role`** — not implemented in oxlint's jsx-a11y plugin. Dropped.
- **Custom `migration-format/migration-format`** rule (was in `eslint-rules/migration-format.js`) — oxlint does not support custom JS rules. Migration-file naming + structure now relies on manual review until either:
  - oxlint adds plugin support, or
  - we revive the rule via a tiny standalone TS check in `scripts/`.
